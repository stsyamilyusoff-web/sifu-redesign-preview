Retain · News list + NewsDetails

A low-traffic but high-trust surface: Sifututor communicates payout schedules, maintenance windows, and feed updates here. Live design uses a flat list with a heavy #E8EFFF unread background and a left-arrow nav; no thumbnails; two different date formats across the app (News uses 24h, Notifications uses 12h). The redesign unifies date format to DD-MMM-YYYY · HH:mm (24h across the whole app), adds the news_image thumbnail (already a live field, unused in the list), softens the unread tint to --brand-weaker, uses a brand dot instead of a coloured card background, and gives the article reader a proper standfirst + feature image with brand-coloured inline links. Tokens only, no em-dashes, light + dark.

Element (live)VerdictProvenance + reasoning
Unread indicator (the dot / icon)CHANGEDLive: two variants called "NewNews" (blue dot) and "OldNews" (grey circle icon), paired with a heavy card-level background (#E8EFFF fill + #DDE7FF border). AFTER: single 8px brand dot for unread rows; read rows get an empty placeholder column (alignment preserved). Unread rows also receive a subtle --brand-weaker tint instead of the heavy blue; the signal is the dot, not the card colour. Cleaner and WCAG-safe.
subject (headline string)KEEPThe live field name is subject. Rendered bold for unread, medium-weight at --ink-2 for read. All five sample articles use it verbatim. Unchanged field, changed visual emphasis.
status (subtitle/teaser)KEEPThe live field name is literally status; it is a subtitle string (not a lifecycle pill). Rendered as a 1-line muted subtitle, truncated with text-overflow:ellipsis. Kept verbatim; label noted in the brief because the field name is confusing.
Timestamp (date + time)CHANGEDLive: DD/MM/YYYY HH:mm (e.g. 30/06/2026 10:05). AFTER: normalised to DD-MMM-YYYY · HH:mm (e.g. 30-Jun-2026 · 10:05). The 24h clock is preserved; only the date format changes to match the single app-wide format (the Notifications screen uses 12h, so News was already inconsistent with it).
news_image (feature thumbnail)ADDA live backend field that is absent from the list view. Added as a 52×52 rounded thumbnail in the trailing zone. Sourced from news_image; drawn here as a tinted CSS block with an emoji glyph and "news_image" label. In the detail view it becomes a full 1.8:1 feature image block.
Left-arrow navigation caretCHANGEDLive: a left-pointing ‹ arrow used as a nav control (non-standard for a list drill). AFTER: replaced with a standard right chevron › (the iOS convention for "drill in"). The tap target and navigation behaviour are unchanged; only the glyph direction is corrected.
content (HTML body, detail only)KEEPRendered as readable prose in the detail reader. Inline links are coloured var(--brand) (previously Google-blue #1a73e8; normalised to brand blue). Two to three paragraphs; bold phrases and links demonstrated in Frame 3.
pre_header (standfirst, detail only)KEEPA live backend field. Rendered as an 18px medium-weight standfirst below the article title and above the timestamp. Gives the reader orientation before scrolling into the body. Shown in Frame 3.
Rich empty stateADDLive empty state was "No News, Yet" plain text. AFTER: a designed empty state (newspaper icon in a brand-tint square, "No news yet", supportive subline). Copy normalised to sentence case. Shown in Frame 2.
Inline links (detail body)CHANGEDLive: links in the content HTML used Google-blue #1a73e8. AFTER: links use var(--brand) (#2B2FBE light / #5B73E8 dark). This is the "link color fixed" note in the screen registry. The underline is softened to rgba(43,47,190,.35) to read as a link without shouting.
1
Scannable feed. The grouped-card list on the grey canvas mirrors the app's established HIG pattern (see Notifications, Help centre). Each row is a self-contained scan unit: dot (unread state) + headline (subject) + teaser (status) + timestamp + thumbnail + chevron. Nothing is dropped; read weight dims the headline to --ink-2 so the eye skips already-read items naturally.
2
Thumbnail from an existing field. news_image exists on the backend news object but has never appeared in the list view. Adding it here adds visual richness and helps tutors distinguish articles at a glance (icon = topic, e.g. green for payouts, amber for maintenance) without any new API work.
3
One date format across the whole app. DD-MMM-YYYY · HH:mm (24h). The live app uses two: News (DD/MM/YYYY HH:mm) and Notifications (hh:mm A DD-MMM-YYYY 12h). This mock normalises News to the single agreed format. The middot separator is consistent with all other timestamp displays in the redesign.
4
Brand links in the article body. The live content field renders raw HTML that historically contained Google-blue anchor tags. The AFTER detail view overrides link colour to var(--brand) globally via a scoped CSS rule on .detail-prose a, so even editor-generated HTML lands on-brand without backend changes. Works in both light and dark via the token cascade.
Commercial pass (Retain · News is a retention channel). Founder: News directly moves tutor retention: payout confirmations reduce churn at payment anxiety moments; new-subject announcements re-activate tutors who stopped applying because their niche seemed absent; maintenance notices prevent frustration and support tickets. Low-frequency but high-trust. Sales: no upsell appropriate here (announcements must be neutral and honest, not promotional); the surface indirectly supports attach-rate by keeping tutors informed about new categories they can apply to. Marketing: a well-designed News reader signals platform professionalism; tutors who trust the comms channel are more likely to participate in referral campaigns announced here. Analyst: instrument "news article open rate" (opens ÷ delivered, per article) and "news tab entry rate" (tutors who visit at least once per fortnight); guard metric = avoid news fatigue (watch unsubscribe or tab-avoidance signals if cadence increases). Baseline: if the feed is read, the redesign should increase open rate vs the flat live list by surfacing thumbnails and a cleaner hierarchy.
