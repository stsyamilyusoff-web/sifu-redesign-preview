Notes for Hafiz
Why this matters: the evaluation report is the parent's first proof and the foundation of the continue-decision. The redesign makes it a report a parent trusts and wants to act on, and ends it on the neutral post-class rating (which builds the tutor's proof, and a low score quietly flags a re-match, the same review loop from the parent wave).
Backend dependencies: (1) render the parent-facing report (the plain layout from the stored answers + the inline level labels; the score shown as a baseline); (2) the rating, already flagged in the parent curated-choice wave. NO new report fields, all 9 are kept, required, and consumed.
Scope: this is the EVALUATION report only (one-time, first class). The PROGRESS report (recurring, monthly) gets the same fast input PLUS carry-forward from last month and a progress trend, its own brief.
Same data the tutor already enters, reframed so the parent receives a clear, personal, forward-looking first-class report that earns the continue, and entered in a fraction of the taps. The trust artifact at the activation moment.
