Deliver · Student Tasks — StudentList · StudentReport · TutorTaskList (2026-07-01)

Three lean list screens sharing the deliver-class-records list grammar.
  • StudentList — who the tutor teaches; rows tap into the existing StudentsDetails screen. Avatars = neutral initials circles, never gendered illustration. Sort by Name / Most subjects.
  • StudentReport — reports already submitted; read-only log. Each row carries: Request UID + student name + subject + report type (Evaluation / Progress) + date. Sort + filter added.
  • TutorTaskList — reports the tutor still owes; action-oriented. Clean inset-card list with a trailing "Submit ›" brand-blue affordance. These are DISTINCT jobs from StudentReport and must stay separate screens.
Colour budget (locked this session): primary CTA = brand-blue (--tint). Progress tag = neutral grey (never green). Evaluation tag = blue tint. Buttons are never green. Colour rides on icon + word, never colour-alone.
Conventions: date DD-MMM-YYYY. Avatars = initials. UID chips are tap-to-copy style. No em-dashes. No invented backend claims. Money omitted here (not a field on these surfaces). No raw hex in frame markup.
Sample data: all names, UIDs, dates are labelled fixtures. Subject counts are indicative.
