Profile · Personal-details editor + Compact Ring tier card (2026-07-01)

Two things live on one pushed screen: (1) the tier-progression hero, using the founder-chosen Compact Ring (Direction B) from profile-tier-card-explore.html — a ~64px circular progress ring stating the fraction/percent once, beside the tier name + next-goal line, replacing the old 3-node Milestone Ladder and its duplicate "N more classes to Pro" callout; (2) the personal-details form (grouped rows, iOS Settings grammar) below the Recommended-next-steps (ProfileGaps) block, ending in a sticky brand-blue Save dock.
Navigation: Profile is a pushed screen (not a tab root) — centered nav title "Profile" + labelled back ("‹ Menu"), never a large title, per the locked hub → detail convention.
Colour budget: primary CTA (Save) = brand-blue, never green. Green = achieved / verified status only (Verified chip, the full All-Star ring + star glyph, the All-Star "complete" success card). Brand-blue = the in-progress ring fill. Colour always rides on icon + word.
Field integrity: every field on the live personal-details form is kept — none dropped. Read-only fields (Full name, Email, Mobile number, NRIC) show a de-emphasised grey value; only Display name is directly editable inline; Gender opens a picker (chevron.up.chevron.down glyph). NRIC only renders when the tutor has one on file, masked as XXXXXX-XX-XXXX.
Conventions: No em-dashes. DD-MMM-YYYY where dates appear. Money = "Earned to date RM 320" is a lifetime earning figure, motivational framing, never a take-home / payout label.
Sample data: Nurul Aisyah (Beginner, progressing state) and Ahmad Zaki (All-Star, complete state) are labelled fixtures, matching the naming used in profile-tier-card-explore.html.
Hafiz data dependencies (flagged, not resolved by this mock): (a) Tier mapping. Backend profile_strength_tier is a 4-value enum (beginner / intermediate / advanced / all_star); this screen assumes a collapse to the founder's 3-tier Beginner / Pro / All-Star vocabulary — confirm the mapping (does intermediate fold into Beginner or Pro?) before wiring. (b) Classes-to-next-tier. Not serialized today — only the tier enum is stored. The ring's "8/20" fraction and "12 classes to Pro" line need the fetch to return current count + next threshold (or a precomputed delta/percent). (c) Earned to date. No lifetime total exists server-side, only monthly cumulative_commission rows. "Earned to date RM 320" is illustrative pending a decision to sum history or add a dedicated lifetime field. (d) Null tier. profile_strength_tier is nullable until edc:recompute-profile-strength runs once — confirm a brand-new tutor renders as Beginner-at-0 (not a blank ring).
