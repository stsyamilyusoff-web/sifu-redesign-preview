Participate · Notifications feed

The screen a tutor reaches from Profile > Notifications (or the badge count). Live today: a flat sectioned list with unread rows highlighted in a heavy #E8EFFF fill + #DDE7FF left border, a 12-hour timestamp, no sender icon chips, and all rows showing a chevron regardless of type. The redesign normalises to a single DD-MMM-YYYY · HH:mm timestamp, adds tinted sender-icon chips (ClassSchedule / EvaluationReport / ProgressReport / Payment / RequestDetail / TutorAwareness) for instant scannability, lightens the unread tint to brand-weaker (subtle, not heavy), and conditionally hides the chevron for TutorAwareness rows (read-only announcements with no drill-down). Tokens only, no em-dashes, light + dark.
Brand blue = brand dot (unread) + brand-tinted chips + actions Green chip = ClassSchedule Amber chip = Payment Grouped-grey canvas Neutral chip = TutorAwareness (no chevron)

Field-level audit (challenge, don't trace · KEEP / CUT / GATE / ADD)

Element (live) Verdict Provenance + reasoning
Unread indicator dot (NewNoti / OldNoti) KEEP  changed Essential at-a-glance read/unread signal. Kept but dot colour normalised: unread = --brand (filled blue), read = --track (neutral grey). Both modes covered.
Title (bold notification headline) KEEP Primary scannable label. Kept bold at 14px, maps directly to the title field on the notification record.
Message body (up to 2 lines) KEEP Supplementary context. Kept at 12.5px muted (--ink-2), clamped to 2 lines so rows do not become walls of text.
Timestamp (live: 12-hour format e.g. "2:30 PM") KEEP  format changed Live app uses 12-hour on Notifications and 24-hour on News. Normalised to the single project format: DD-MMM-YYYY · HH:mm (24-hour). The date component adds calendar context for older items.
Recent / Older grouping (section headers) KEEP The temporal grouping is genuinely useful: it separates actionable new items from historical ones. Section labels styled with the shared .seclabel token (uppercase, muted, small).
Unread row tint (live: #E8EFFF fill + #DDE7FF left border) KEEP  lightened The tint communicates "action pending here" and must stay. Heavy #E8EFFF hardcoded hex replaced with --brand-weaker (6% brand opacity, light; 10% in dark). Softer and token-correct; the dot remains the primary signal so the row tint is a reinforcing hint, not a shout.
Chevron (right arrow, all rows in live) KEEP  conditional Kept on all drill-down rows (ClassSchedule, EvaluationReport, ProgressReport, Payment, RequestDetail). Conditionally hidden when sender == 'TutorAwareness': these are read-only broadcast announcements with no detail screen. Showing a chevron on an untappable row is a lie in the UI; the live app does this (Welcome row shows faded chevron). Fixed in the redesign.
Sender icon chip (not present in live) ADD The sender field already exists on every notification record (ClassSchedule / EvaluationReport / ProgressReport / Payment / RequestDetail / TutorAwareness). Mapping it to a tinted emoji chip (calendar, document, money, briefcase, chart, megaphone) gives the tutor a visual category at a glance before reading the title. No new data required.
Rich empty state (not present in live) ADD When the list is empty the live screen shows nothing. The redesign adds a real designed state: bell icon block + "No notifications yet" headline + supportive subline naming what will appear. Prevents a confusing blank screen for new tutors.
1
Sender icon chips added for instant scannability. The sender field is already populated on every notification. Mapping it to a small tinted chip (calendar = ClassSchedule, document = reports, briefcase = RequestDetail, money = Payment, megaphone = TutorAwareness) lets a tutor scan by category before reading. Zero new backend data required.
2
Single date format, everywhere. The live app uses 12-hour on this screen and 24-hour on the News screen. All timestamps now follow DD-MMM-YYYY · HH:mm. The full date matters for Older section items (a time alone like "9:00 AM" is meaningless for a June entry seen in July).
3
TutorAwareness rows carry no chevron. These are platform-broadcast messages (Welcome, policy changes, feature announcements) with no detail drill-down. The live app already dims the chevron on the Welcome row, acknowledging it is not tappable. The redesign removes it entirely for sender == 'TutorAwareness'. Every other sender type (ClassSchedule, EvaluationReport, ProgressReport, Payment, RequestDetail) keeps its chevron because it links to a real detail screen.
4
Unread tint lightened; dot remains the primary signal. The live heavy blue fill (#E8EFFF + #DDE7FF border) dominates the eye so aggressively it overwhelms the title text. The redesign replaces it with --brand-weaker (6% brand opacity), a breath of tint that reinforces what the blue dot already signals. Both modes: the CSS var shifts automatically to the dark-mode equivalent (10% brand).
Commercial pass (re-engagement channel). Founder: Notifications are the primary pull-back mechanism for tutors who have gone quiet. A tutor who sees a timely "New request matches your subjects" is the top-of-funnel re-activation moment; every friction point (ambiguous unread treatment, missing context chip, broken chevron on announcements) is a reason to dismiss the screen. The redesign removes those frictions. Sales: Payment and RequestDetail notifications are the two highest-intent touchpoints (money arrived / new work available). Surfacing sender chips lets the tutor prioritise those rows in one glance without reading every line. No upsell pattern here; the integrity of the channel matters more than a conversion nudge. Marketing: The notification feed is the operational email replacement; tutors who trust it stay in-app rather than hunting for WhatsApp. Rich, accurate notifications (correct format, correct chevron logic) build that trust over time. The empty state sets an honest expectation for new tutors. Analyst: Instrument notification open rate per sender type (which category actually drives taps), and the drop-off between notification received and the linked detail screen opened. Guard metric: notification dismissal rate should fall after the unread tint lightening (a harsh tint that covers the whole row can read as an error state and trigger dismissal).
