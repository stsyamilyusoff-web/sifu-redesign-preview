Tally: KEEP 16 · ADD 3 · CUT 3 (all 3 gated on the new billing model / allocator flag). Every scheduling safeguard kept. The only removals are the quota cap, the extra-class path, and the quota-derived status copy. *Repeat-weekly kept but re-bounded off remaining_classes onto the calendar window (see dependency note).
Removed (and why each is safe): (1) Quota-cap block: in the live model a session past remaining_classes could not be scheduled and pushed you to the extra-class request. The allocator now auto-invoices each scheduled class, so capping scheduling would block real teaching that should simply be billed. (2) Extra-class path (ExtraClass + SubmitRequestExtraClass): its only job was requesting paid sessions beyond quota; the auto-invoice makes the request redundant. (3) The 7 emoji status sentences: they narrated the quota count ("schedule the remaining 3 classes", "paid sessions from last month"); with no cap they are inaccurate and chatty. All three are gated on the allocator flag and ship with the cutover, not before.

Commercial pass (Rule 16)

Founder: in the new model every scheduled class drives an auto-invoice, so this form sits directly on the revenue line. Its job is to make scheduling fast and correct (more classes scheduled = more GMV recognised), not to gate it. Highest leverage = removing the cap friction that previously sent tutors into a dead-end request.
Sales: no upsell surface here (utility form); the auto-invoice IS the monetisation, happening silently and correctly. The honest move is the neutral "Each class you schedule is invoiced automatically" line so the tutor (and downstream the parent) is never surprised by a charge.
Marketing: "schedule in seconds, billing handled for you" is the tutor-facing promise; the removed cap + removed extra-class detour is a real friction win to message in release notes.
Analyst: primary metric = classes scheduled per active tutor / week (expect a lift once the cap is gone) and schedule-to-invoice accuracy. Guard metric (non-negotiable): no accidental over-scheduling / wrong-date classes. Because each row now bills, an over-scheduled or mis-dated class is a billing error. The guard is held by keeping every safeguard (clash check, the backdate cliff, the explicit confirm), plus the review-before-confirm summary. Instrument: rate of invoiced classes later cancelled/refunded as wrong-date, and backdate-escalation volume.
