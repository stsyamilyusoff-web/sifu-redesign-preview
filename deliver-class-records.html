Deliver · Class Records — list · filter · detail (2026-07-01)

Three screens, one shared "class card" visual. The class card is the single source of truth for identity: it carries student + subject/level + RM + icon-row ABOVE the separator, and a footer row (ID chip left · status badge right) BELOW it — in both the list and the detail hero. In the list, each card reads top-to-bottom: [student · RM] / [subject · level] / [date · duration · location · mode] / separator / [CLS-XXXXXX · status]. In the detail, the hero footer carries ID chip + status badge (with an admin-date caption for Verified). Sections below show only non-duplicated facts: Student = name only; Schedule = time-range only; Attendance = clock in / out / time taught. The filter stays its own screen: a grouped-list form (iOS Settings grammar) that overlays the list on demand, keeping the list surface uncluttered.
Colour budget (locked this session): primary CTA = brand-blue (--tint), never green. Green = Attended / Verified status. Amber = the one check-this flag (Dispute / Cancelled). Scheduled = neutral. No red (not in the budget). Colour always rides on icon + word, never colour-alone.
Conventions: date format DD-MMM-YYYY throughout. Money = "RM 94" on list scan cards, real cents on the detail ("RM 94.00"). No em-dashes. No invented backend claims. Gender not shown (not relevant here). No raw hex in frame markup. No fact appears twice on any screen.
Sample data: all names, amounts, UIDs, dates are labelled fixtures. Backend field names are noted in comments.
Status vocabulary: the authoritative ClassStatus.php enum = Scheduled · Ongoing · Attended · Verified · Cancelled · Postponed · Refunded (no "pending"). Dispute is a frontend display value (the live ClassRecords list renders it) but is NOT in that enum — flagged to Hafiz.
