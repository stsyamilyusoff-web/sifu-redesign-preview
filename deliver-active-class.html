Deliver · Active class cockpit — rich mock (2026-06-29)

The single most-used Win & Deliver screen: every class passes through it, and it gates two funnel metrics (on-time clock-in, classes delivered). The live screen is a context-free timer blob ("Click on the Timer to Clock Out") with no who / what / where, no earning, no progress, and an undiscoverable tap-the-timer affordance.

What the redesign keeps (the solid parts the brief protects): photo proof at clock-in and clock-out, the running timer, and crash-recovery (the timer reads from the server start-time, so a reopened app resumes — shown in the In-progress frame's "resumed" note). What it adds: (1) a class-context card (student · subject · level · location · scheduled time · this-class earning) so the tutor always knows the class they are in; (2) a scheduled vs elapsed progress bar; (3) an explicit, labelled primary button replacing the hidden tap-the-timer gesture; (4) a state banner carrying the one state colour; (5) a done summary with the earning and a one-tap path to the post-class report (closing the Deliver → Report loop). Retired: legacy ClockIn / ClockOut / ClassTimerCount standalone screens fold into these states.

Grounded in the gig-worker in-job cockpit (Grab / Uber driver: who + where + one big action + completion proof) and timer apps (Apple Clock / Toggl: large tabular-numeral time as the focal element). Colour budget respected: green = live / done / GPS-confirmed, amber = one caution (early / GPS-off), everything else neutral. Tokens only, no raw hex in markup. Sample data is labelled; backend dependencies are listed at the foot of the page.

Founder review (2026-06-29): cockpit APPROVED.
