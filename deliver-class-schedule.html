Deliver · Class schedule cockpit — rich mock (2026-06-30)

The schedule is the tutor's daily home and the top retention lever after ActiveClass: a tutor who opens it, sees their day at a glance, and walks into the next class easily comes back and applies more. Honest read after re-grounding on the live populated screen: the live app already has a week strip + a vertical timeline + reasonably rich class cards, so this is enrichment, not a rebuild. The gaps are: no today-summary, no week-level density, a colour-budget violation (green In-Person pill + green status both shout), a noisy radio-icon status, and a dead empty state ("No Class, Yet" + a big illustration + nothing actionable).

What changes: (1) a today-summary ("4 classes · RM 72" by the date); (2) density dots on the week strip so the tutor sees load at a glance + a this-week count footer; (3) the Ongoing "now" class gets a green accent + an inline "Clock out" CTA that jumps straight into the ActiveClass cockpit (closing schedule -> class loop); (4) governed status chips on the real lifecycle (see below) instead of two competing greens; (5) a working empty state (add a class · why it matters) and a per-day empty row for a free day inside a populated week. Mode + location are neutral (colour budget).

Schedule focuses on the CLASS, not money (2026-06-30, founder feedback): the per-class RM earning is removed from the cards, the today-summary, and the week footer — earnings live in the Earn cluster, not here. The schedule shows just the class facts and class counts. The week footer reads "12 classes" and the today-summary "4 classes".

Time handling (gutter removed, 2026-06-29; card restructured, 2026-06-30): the narrow left time-gutter is removed — it ate horizontal width and crammed the labels onto two wrapped lines, squeezing the cards. Time now lives on the card: each card opens with a full-width header — full range + duration ("2:00 – 3:00 PM · 1h") on the left, mode (🏠 In-person / 💻 Online) + the governed status chip on the right — so there is no empty right side and the card body gets the full frame width. A slim full-width time-group header (state node + "2:00 PM" + hairline) sits above each slot for at-a-glance scanning, replacing the gutter without pushing the cards right. Overlapping / back-to-back classes are grouped under one slot header with a "Back-to-back" tag and a shared connector hairline, so two same-window classes read as one tight set instead of overlapping time labels.

Week strip alignment (2026-06-30, founder feedback): all 7 day cards share one shape and height; today is a simple brand-tint highlight (fill + tint border + tint number) on the same card, not a taller black widget. The dots row has a fixed height so day labels and numbers stay aligned across days with and without density dots.

Expandable week-to-month calendar (2026-06-30, founder-approved enhancement): the week strip now pulls down to a full month grid (the ExpandableCalendar pattern). Collapsed = week = the existing 7-day strip, unchanged, with a small drag-handle + "Tap to see the month ▾" affordance below it. Expanded = month (the new frame) = a "June 2026" card with a S M T W T F S header and a 7-column date grid; each day cell carries the same density dots as the week strip (1 to 4 = class count), today gets the brand-tint highlight, the selected day a solid tint fill, and a "Tap to collapse to the week ▴" handle sits under the grid. Below the grid: the selected day's compact list (its slim Attended row + the Ongoing now-card). This is an enhancement — the live app already has the month context (react-native-month-year-picker + getCurrentMonthDates in Screens/ClassSchedule/index.tsx) and a month picker, but no grid view; the per-day density comes from the same schedule data the week strip already reads, nothing faked. Today/selected use the brand tint (green stays reserved for live/fit). Note: the grid uses the mock's internal sample calendar where 24 = Tuesday, to stay consistent with the locked week strip above it; a real build would derive the weekday layout from the actual month.

Cards rebuilt on the REAL ClassStatus lifecycle (2026-06-30, founder-approved): grounded in the backend enum app/Enums/ClassStatus.php, not invented states. The authoritative 7 statuses are Scheduled · Ongoing · Attended · Verified (the happy path) plus the edges Cancelled · Postponed · Refunded. Lifecycle: Scheduled -> Ongoing -> Attended -> Verified (with Scheduled/Ongoing -> Cancelled, and Cancelled -> Refunded). The day now shows that lifecycle across time: a finished earlier class is Attended (clocked out, awaiting verification) — shown as a slim, de-emphasized one-line row since a done class is "for reference" and earns the lightest weight (DR-002 density grammar); the current class is Ongoing (the locked now-card treatment + Clock-out CTA, grounded in the live app's ongoing -> clock_out action); the upcoming classes are Scheduled (with the live app's Postpone + Attend actions on the next-up card); and the past-day frame's class is Verified (a past class now confirmed, the truthful end-state). The fake "Pending parent confirm" card is removed — "pending" has no backing in the enum (the live app still carries a dead == 'pending' check in its status filter + status-dot switch, alongside non-enum 'approved'/'rejected' cases).

OPEN QUESTION — status-chip colour reconciliation (not decided here): the backend enum assigns each status its own colour — Scheduled=blue · Ongoing=amber · Attended=purple · Verified=green · Cancelled=red · Postponed=amber · Refunded=slate. These conflict with our colour budget (where green = live/fit only, amber = a single "check this" flag). For now every status chip is held as a governed NEUTRAL grey pill (only the Ongoing now-card keeps the green live indicator, which is the locked now-card treatment, not the enum's green). We are not adopting the backend colours and not claiming this neutral mapping is final — the reconciliation (do we recolour the chips to the enum, keep neutral, or define a budget-safe subset?) is an open decision for the founder.

Grounded in the gig-worker "today" view (Grab / Uber driver daily plan: today's jobs up top) and calendar day-agendas (Apple / Google Calendar timeline + Fantastical's day list), and in the live app's per-state actions (Screens/ClassSchedule/index.tsx: Scheduled = Postpone/Attend, Ongoing = Clock out, past/ongoing = Mark attended). Tokens only, no raw hex in app UI. Sample data labelled; backend deps at the foot.
